Accessibility Samplers

Findings

  • A good article by Kathleen McMahon of 24 Accessibility (Pixels vs. Relative Units in CSS: why it’s still a big deal, 2019) reviews the reasons that we must use unitless values in our CSS. The author's experience mirrors mine; there are conflicting opinions about how to implement responsive typography, and sources that I consider to be good ones aren't in alignment. Says McMahon:

    When I first started looking up recommendations on whether or not using pixels in CSS in general is a good idea, I’d find conflicting opinions on the subject, such as articles like this one describing why you should just use pixels or should stop using pixels in CSS. [This article by Zell] ...about responsive sizing in media queries [which argues to use em, not pixels, for media queries]. Even a good article about Accessibility in Resizing Text about the behavior of older browsers.

    There wasn’t much mention — or consideration — that users really do dig into the advanced browser settings to change their default font size.

  • Several days ago I started out by using the following calculation for font size, because Heydon Pickering is a reasonable source to use. See sketch 1; he proposes using the following code:

    html { font-size: calc(1em + 1vw); }

    In my test, body text on a sample page (accessibility sketch #1) is displayed at 28px.

    And if I adjust the browser's zoom setting the text size on the web page adjusts, so presumably the 1em part of the calculation overcomes the otherwise problematic (potentially inaccessible) vw part of the calculation? But I have cold feet about using vw as part of the calculation...and I can see that failing to use clamp makes things get too big at larger viewport widths.

  • And then I never got around to troubleshooting this example from Jason Pamental's presentation at the 2019 Beyond Tellerrand conference ("Dynamic typography with variable fonts"). This doesn't seem to have any effect and I get a css validation error that i can't solve:

    /* Please scale from 2.25em on small screens up to 5em on largest ones, starting at vw of 25em and stopping at 75em or larger. */

    font-size: calc([size-min]em + ([size-max] - [size-min]) * ((100vw - [start]em) / ([stop] - [start])));

  • In his Viewport Unit Based Typography article (9 March 2016), Zell Liew proposes setting a minimum font-size, then scaling the font with a small viewport multiple by using the calc() property:

    html { font-size: calc(18px + 0.25vw) }

    Zell says he learned about this technique through Mike Riethmuller’s 2015 Precise control over responsive typographyarticle. Unfortunately, he says that the code doesn’t work on some browsers. Specifically, it doesn’t resize on Safari for Mac, and I don't see updated information about whether this is still true.

    The fix, says Zell, is to combine the use of a percentage unit with viewport width to allow Safari to scale the fonts:

    html { font-size: calc(112.5% + 0.5vw) }

    I'm really confused about whether the use of vw to calculate font size breaks accessibility guidelines. Does this work okay if a user has low vision and resets the default browser font size?

  • I figured I needed to focus on clamp in order to eliminate problems at small and large viewport sizes. I liked the simplicity of what designcourse.com advises in a fluid typography video, but he uses both pixel and viewport width measurements along with ems, and I don't think that is appropriate. Right?

    h1 { font-size: clamp(40px, 5vw, 70px); }
    p { font-size: clamp(1em, 2vw, 2em); }

  • Yet the clamp spec. itself uses pixels:

    html {
    font-family: sans-serif;
    }

    body {
    margin: 0 auto;
    width: min(1000px, calc(70% + 100px));
    }

    h1 {
    letter-spacing: 2px;
    font-size: clamp(1.8rem, 2.5vw, 2.8rem);
    }

    p {
    line-height: 1.5;
    font-size: max(1.2rem, 1.2vw);
    }

Conclusion: I'm still confused. The above example in the spec appears to work okay when I adjust the browser's default font size. For now I'm going to use the format shown in the spec as my template, adjusting the values as needed, until I land on a different or better solution. When in doubt, go with the spec. I know that clamp is the correct approach because of the control it gives me as the lower and higher ranges, and I'll learn more as I go along.
